Documentation Deployment

The repository contains a GitHub Actions workflow at .github/workflows/docfx-deploy.yml.

On pushes to master, it:

  1. checks out this repository;
  2. installs DocFX;
  3. builds docs/docfx.json;
  4. checks out the public documentation repository;
  5. copies docs/_site into that repository;
  6. commits and pushes the generated documentation.

The workflow uses the DOCS_DEPLOY_KEY secret to push to the public docs repository.

Generated Documentation Folders

These paths are generated and ignored by git:

  • docs/_site;
  • docs/.cache;
  • docs/api;

Rebuild them with:

docfx .\docs\docfx.json

Catalog Database Startup

At application startup, SeedCatalogDbAsync checks whether the catalog database exists, creates it when necessary, applies catalog migrations, and logs whether tenant records are present.

Tenant Database Lifecycle

Tenant database operations are performed through infrastructure services:

Operation Service
Create tenant database TenantDatabaseManager
Apply tenant migrations TenantMigrator
Provision a tenant TenantProvisioningWorkflow
Drop tenant database TenantDatabaseManager

Deletion Policy

Hard deletion requires soft deletion first. This protects active tenants from accidental database drops and gives the catalog a clear deletion trail before the physical tenant database is removed.
